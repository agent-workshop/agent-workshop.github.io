# LLMを使ったテスト自動生成：実装例とハマりポイント

## 導入

ソフトウェア開発において、テストコードの記述は避けて通れない作業です。特に単体テストは品質保証の要ですが、カバレッジを意識し始めると「似たようなテストケースを大量に書く」という単調な作業が発生しがちです。

LLM（大規模言語モデル）を活用することで、テストコードの生成を自動化できます。具体的には以下のようなメリットが期待できます。

- 正常系・異常系の網羅的なテストケース生成
- エッジケースの洗い出し
- テストコードのひな形生成による工数削減

ただし、LLMが生成するテストコードには品質のばらつきがあり、そのまま使うと問題が発生することも少なくありません。本記事では、実際に動作するコード例とともに、実装時の注意点を解説します。

## 前提条件

本記事のコードは以下の環境で動作確認しています。

- Node.js 18以上
- TypeScript 5.0以上
- テストフレームワーク: Vitest（Jestでも同様に動作します）
- LLM API: OpenAI API（gpt-4o-miniを使用）
- ライブラリ: `openai`（v4系）

```bash
npm install openai vitest
```

## 実装手順

### ステップ1: LLMクライアントの準備

まず、LLMにテストコードを生成させるためのクライアントを作成します。環境変数からAPIキーを読み込むようにしておきます。

```typescript
// llmClient.ts
import OpenAI from 'openai';

const client = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
});

export async function generateTestCode(
  sourceCode: string,
  functionName: string
): Promise<string> {
  const response = await client.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      {
        role: 'system',
        content: `あなたはテストコード生成の専門家です。
以下のルールに従ってテストコードを生成してください。
- TypeScriptのVitest形式で出力する
- describe/itを使用する
- 正常系と異常系を両方含める
- エッジケースも考慮する
- モックは最小限にする
- コードブロックのみを出力する（説明は不要）`,
      },
      {
        role: 'user',
        content: `以下の関数のテストコードを生成してください。

\`\`\`typescript
${sourceCode}
\`\`\`

対象の関数名: ${functionName}`,
      },
    ],
    temperature: 0.3,
    max_tokens: 2000,
  });

  const content = response.choices[0]?.message?.content;
  if (!content) {
    throw new Error('Failed to generate test code');
  }

  // コードブロックからコード部分のみを抽出
  const codeMatch = content.match(/```typescript\n([\s\S]*?)```/);
  return codeMatch ? codeMatch[1].trim() : content.trim();
}
```

### ステップ2: テスト対象の関数を定義する

例として、シンプルなユーザー情報バリデーション関数をテスト対象とします。

```typescript
// userValidator.ts
export interface UserInput {
  name: string;
  email: string;
  age: number;
}

export function validateUser(input: UserInput): { valid: boolean; errors: string[] } {
  const errors: string[] = [];

  if (!input.name || input.name.trim().length === 0) {
    errors.push('名前は必須です');
  } else if (input.name.length > 50) {
    errors.push('名前は50文字以内である必要があります');
  }

  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  if (!input.email || !emailRegex.test(input.email)) {
    errors.push('有効なメールアドレスを入力してください');
  }

  if (input.age === undefined || input.age === null) {
    errors.push('年齢は必須です');
  } else if (input.age < 0 || input.age > 150) {
    errors.push('年齢は0〜150の範囲で入力してください');
  } else if (!Number.isInteger(input.age)) {
    errors.push('年齢は整数で入力してください');
  }

  return {
    valid: errors.length === 0,
    errors,
  };
}
```

### ステップ3: テスト生成と実行を自動化するスクリプト

生成したテストコードをファイルに書き出し、自動実行するスクリプトを作成します。

```typescript
// generateTests.ts
import fs from 'fs';
import path from 'path';
import { generateTestCode } from './llmClient';

async function main() {
  const sourceFile = process.argv[2];
  if (!sourceFile) {
    console.error('使用方法: npx tsx generateTests.ts <対象ファイル>');
    process.exit(1);
  }

  const sourceCode = fs.readFileSync(sourceFile, 'utf-8');
  const functionName = 'validateUser'; // 対象関数を指定（実際のプロジェクトでは動的に抽出）

  console.log('テストコードを生成中...');
  const testCode = await generateTestCode(sourceCode, functionName);

  const testFilePath = sourceFile.replace('.ts', '.test.ts');
  fs.writeFileSync(testFilePath, testCode, 'utf-8');
  console.log(`テストファイルを作成しました: ${testFilePath}`);
}

main().catch(console.error);
```

実行方法:
```bash
npx tsx generateTests.ts userValidator.ts
```

## コードサンプル

### サンプル1: 生成されたテストコードの例

実際に上記のスクリプトで生成されたテストコードの例です。

```typescript
// userValidator.test.ts
import { describe, it, expect } from 'vitest';
import { validateUser } from './userValidator';

describe('validateUser', () => {
  // 正常系
  it('有効なユーザー情報に対してtrueを返す', () => {
    const result = validateUser({
      name: '山田太郎',
      email: 'taro@example.com',
      age: 30,
    });
    expect(result.valid).toBe(true);
    expect(result.errors).toHaveLength(0);
  });

  // 異常系: 名前が空
  it('名前が空の場合エラーを返す', () => {
    const result = validateUser({
      name: '',
      email: 'taro@example.com',
      age: 30,
    });
    expect(result.valid).toBe(false);
    expect(result.errors).toContain('名前は必須です');
  });

  // 異常系: 無効なメールアドレス
  it('無効なメールアドレスでエラーを返す', () => {
    const result = validateUser({
      name: '山田太郎',
      email: 'invalid-email',
      age: 30,
    });
    expect(result.valid).toBe(false);
    expect(result.errors).toContain('有効なメールアドレスを入力してください');
  });

  // エッジケース: 年齢が負の値
  it('年齢が負の値の場合エラーを返す', () => {
    const result = validateUser({
      name: '山田太郎',
      email: 'taro@example.com',
      age: -1,
    });
    expect(result.valid).toBe(false);
    expect(result.errors).toContain('年齢は0〜150の範囲で入力してください');
  });

  // エッジケース: 年齢が小数
  it('年齢が小数の場合エラーを返す', () => {
    const result = validateUser({
      name: '山田太郎',
      email: 'taro@example.com',
      age: 25.5,
    });
    expect(result.valid).toBe(false);
    expect(result.errors).toContain('年齢は整数で入力してください');
  });
});
```

### サンプル2: 複数の関数に対応する改良版

より実用的にするため、ソースファイルから関数定義を抽出する処理を追加します。

```typescript
// generateTestsImproved.ts
import fs from 'fs';
import { generateTestCode } from './llmClient';

function extractFunctionNames(sourceCode: string): string[] {
  // 簡易的な関数名抽出（実際のプロジェクトではASTパーサーを使うことを推奨）
  const regex = /export\s+function\s+(\w+)/g;
  const names: string[] = [];
  let match;
  while ((match = regex.exec(sourceCode)) !== null) {
    names.push(match[1]);
  }
  return names;
}

async function generateTestsForFile(sourceFile: string) {
  const sourceCode = fs.readFileSync(sourceFile, 'utf-8');
  const functionNames = extractFunctionNames(sourceCode);

  for (const funcName of functionNames) {
    console.log(`テスト生成中: ${funcName}`);
    const testCode = await generateTestCode(sourceCode, funcName);

    const testFilePath = sourceFile.replace('.ts', `.${funcName}.test.ts`);
    fs.writeFileSync(testFilePath, testCode, 'utf-8');
    console.log(`作成: ${testFilePath}`);
  }
}

// 使用例
const targetFile = process.argv[2];
if (targetFile) {
  generateTestsForFile(targetFile).catch(console.error);
}
```

## ハマりやすいポイント

### 1. LLMが生成するコードの構文エラー

LLMの出力は必ずしも完璧なTypeScriptコードとは限りません。特に以下のような問題が発生しやすいです。

- インポート文の欠落や誤り
- 型定義の不整合
- 存在しないメソッドの呼び出し

**対策**: 生成後に`tsc --noEmit`で型チェックを自動実行し、エラーがあれば再生成する仕組みを組み込みましょう。

### 2. テストの過剰生成

LLMはプロンプトで指定した以上に多くのテストケースを生成する傾向があります。特に「網羅的に」と指示すると、数百行のテストコードが生成されることがあります。

**対策**: `max_tokens`を適切に設定し、プロンプトで「重要なケースに絞って5〜10個のテストを生成してください」と明示します。

### 3. 外部依存のモック漏れ

データベースや外部APIに依存する関数の場合、LLMは適切なモックを生成しないことがあります。結果として、テスト実行時に実際の外部サービスにリクエストが飛んでしまう危険性があります。

**対策**: プロンプトに「外部依存がある場合はvitestのvi.mockを使ってモックしてください」と明示的に指示を追加します。

### 4. プロンプトエンジニアリングの難しさ

同じソースコードでも、プロンプトの書き方で生成結果が大きく変わります。特に日本語のプロンプトと英語のプロンプトで品質差が出る場合があります。

**対策**: いくつかのプロンプトパターンを用意し、生成結果を比較検証する仕組みを作ると安定します。

### 5. トークン消費とコスト

大規模なソースコードを毎回LLMに送信すると、APIのトークン消費が大きくなります。ファイル全体を送るのではなく、関数単位で切り出して送信することでコストを抑えられます。

## まとめ

LLMを使ったテスト自動生成は、単調なテストコード記述の負担を軽減できる有効な手段です。ただし、生成されたコードをそのまま信用せず、以下の点を意識することが重要です。

- 構文チェックと型チェックを自動化する
- プロンプトを適切に設計して品質をコントロールする
- 外部依存のモック漏れに注意する
- コストと品質のバランスを考慮する

本記事で紹介した実装をベースに、プロジェクトの特性に合わせてカスタマイズしてみてください。テスト自動生成を完全自動化するのではなく、アシスタントとして活用することで、開発効率の向上に役立てられます。
