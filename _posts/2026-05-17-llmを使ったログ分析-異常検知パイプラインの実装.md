# LLMを使ったログ分析：異常検知パイプラインの実装

## 1. 導入

システム運用において、ログ分析は障害の早期発見や原因特定に欠かせない工程です。しかし、大量のログを人力で監視するには限界があり、従来のルールベースの監視では想定外の異常パターンを見逃しやすいという課題があります。

LLM（大規模言語モデル）をログ分析に活用することで、以下のようなメリットが得られます。

- 固定ルールでは検出しづらい、文脈に依存した異常を発見できる
- ログの自然言語的な意味を理解し、エラーの原因を推測できる
- 新しいパターンの異常にも柔軟に対応できる

本記事では、TypeScriptを使用して、LLMを組み込んだログ異常検知パイプラインを実際に構築する手順を解説します。

## 2. 前提条件

### 環境
- Node.js 18以上
- npm または yarn
- OpenAI API キー（または互換性のあるLLM API）

### 使用ライブラリ
- `openai`（公式Node.js SDK）
- `winston`（ログ管理）
- `dayjs`（日付処理）

インストールは以下のコマンドで行います。

```bash
npm install openai winston dayjs
```

## 3. 実装手順

### ステップ1: ログ収集と前処理

まず、アプリケーションログを収集し、LLMで分析しやすい形式に整形します。ここではwinstonを使って標準出力とファイルにログを出力し、その中からエラーレベルのログを抽出します。

### ステップ2: LLMによる異常判定

抽出したログをLLMに送信し、異常の有無とその理由を判定させます。プロンプト設計が重要で、具体的な指示を与えることで誤検知を減らせます。

### ステップ3: 結果の保存と通知

判定結果をデータベースやファイルに保存し、必要に応じて通知を行います。

## 4. コードサンプル

### サンプル1: ログ収集と前処理モジュール

```typescript
import winston from 'winston';
import dayjs from 'dayjs';

// ロガーの設定
const logger = winston.createLogger({
  level: 'info',
  format: winston.format.json(),
  transports: [
    new winston.transports.File({ filename: 'app.log' }),
    new winston.transports.Console()
  ]
});

// ログ収集関数（実際のアプリケーションではミドルウェアなどで呼び出す）
export function logEvent(level: string, message: string, meta?: Record<string, unknown>) {
  logger.log({ level, message, timestamp: dayjs().format('YYYY-MM-DD HH:mm:ss'), ...meta });
}

// 直近のエラーログを取得（簡易的な実装）
export function getRecentErrorLogs(hours: number = 1): Promise<string[]> {
  return new Promise((resolve) => {
    const errors: string[] = [];
    const stream = require('fs').createReadStream('app.log', { encoding: 'utf-8' });
    
    stream.on('data', (chunk: string) => {
      const lines = chunk.split('\n').filter(line => line.includes('"level":"error"'));
      errors.push(...lines);
    });

    stream.on('end', () => {
      // 直近hours時間のログのみ抽出（簡易版）
      const cutoff = dayjs().subtract(hours, 'hour').unix();
      const filtered = errors.filter(line => {
        const match = line.match(/"timestamp":"([^"]+)"/);
        if (!match) return false;
        return dayjs(match[1]).unix() >= cutoff;
      });
      resolve(filtered.slice(-50)); // 最大50件
    });
  });
}
```

### サンプル2: LLM異常検知パイプライン

```typescript
import OpenAI from 'openai';

const openai = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY
});

// 異常検知のためのプロンプト
const ANOMALY_DETECTION_PROMPT = `
あなたはシステムログの異常検知アシスタントです。
以下のログ群を分析し、以下の観点で評価してください。

1. 明らかなエラー（Exception, Error, Failedなど）
2. 通常運用では発生しない異常なパターン（短時間の大量リクエスト、不自然な値など）
3. セキュリティ上の懸念（認証失敗の連続、不正なアクセスパターン）

出力形式は以下のJSONのみとし、余計な説明は不要です。
{
  "has_anomaly": boolean,
  "anomaly_count": number,
  "anomalies": [
    {
      "severity": "critical" | "warning" | "info",
      "log_line": "該当ログの内容",
      "reason": "異常と判断した理由",
      "suggestion": "推奨される対応"
    }
  ]
}
`;

interface AnomalyResult {
  has_anomaly: boolean;
  anomaly_count: number;
  anomalies: Array<{
    severity: 'critical' | 'warning' | 'info';
    log_line: string;
    reason: string;
    suggestion: string;
  }>;
}

export async function detectAnomalies(logLines: string[]): Promise<AnomalyResult> {
  if (logLines.length === 0) {
    return { has_anomaly: false, anomaly_count: 0, anomalies: [] };
  }

  const response = await openai.chat.completions.create({
    model: 'gpt-4o-mini', // コストを抑えたい場合はgpt-4o-mini推奨
    messages: [
      { role: 'system', content: ANOMALY_DETECTION_PROMPT },
      { role: 'user', content: `分析対象のログ:\n${logLines.join('\n')}` }
    ],
    temperature: 0.1, // 一貫性を高めるため低めに設定
    response_format: { type: 'json_object' }
  });

  const content = response.choices[0]?.message?.content;
  if (!content) {
    throw new Error('LLMからの応答が空でした');
  }

  return JSON.parse(content) as AnomalyResult;
}
```

### サンプル3: パイプラインの実行例

```typescript
import { logEvent, getRecentErrorLogs } from './logCollector';
import { detectAnomalies } from './anomalyDetector';

async function runPipeline() {
  try {
    // 1. ダミーのエラーログを生成（実際の運用ではアプリケーションのログを使用）
    logEvent('error', 'Database connection timeout after 5000ms', { service: 'db', retry: 3 });
    logEvent('error', 'HTTP 500 on /api/users: Internal Server Error', { endpoint: '/api/users' });
    logEvent('warn', 'High memory usage: 85%', { threshold: 80 });

    // 2. 直近のエラーログを取得
    const recentLogs = await getRecentErrorLogs(1);
    console.log(`取得したログ数: ${recentLogs.length}`);

    // 3. LLMで異常検知
    const result = await detectAnomalies(recentLogs);

    // 4. 結果を出力
    if (result.has_anomaly) {
      console.log(`異常検出: ${result.anomaly_count}件`);
      result.anomalies.forEach(a => {
        console.log(`[${a.severity.toUpperCase()}] ${a.reason}`);
        console.log(`  ログ: ${a.log_line}`);
        console.log(`  提案: ${a.suggestion}`);
      });
    } else {
      console.log('異常は検出されませんでした');
    }
  } catch (error) {
    console.error('パイプライン実行エラー:', error);
  }
}

// 1時間ごとに実行（cronなどで定期実行）
runPipeline();
```

## 5. ハマりやすいポイント

### プロンプト設計の重要性
LLMはプロンプトの内容に大きく影響されます。以下の点に注意してください。

- **温度パラメータ**：異常検知のようなタスクでは`temperature: 0.1`程度の低い値が適切です。高すぎると毎回異なる判断をします。
- **出力形式の指定**：JSON形式を強制することで、後続処理が安定します。`response_format: { type: 'json_object' }`を指定すると、OpenAIのモデルがJSONを確実に返します。
- **プロンプトの具体性**：「異常を検出してください」だけでは曖昧です。具体的な観点（エラー、パターン、セキュリティ）を列挙しましょう。

### トークン数とコスト
ログの量が多いとAPIコールのトークン数が増加し、コストが高くなります。以下の対策が有効です。

- 分析対象を直近のログに限定する（例：過去1時間、最大50件）
- エラーレベルのログのみ抽出する
- 同一内容のログは集約してから送信する

### レート制限とリトライ
OpenAI APIにはレート制限があります。大量のログを短時間に分析する場合は、以下の対応が必要です。

```typescript
// 簡易的なリトライ機構
async function withRetry<T>(fn: () => Promise<T>, retries = 3): Promise<T> {
  for (let i = 0; i < retries; i++) {
    try {
      return await fn();
    } catch (error) {
      if (i === retries - 1) throw error;
      const delay = Math.pow(2, i) * 1000; // 指数バックオフ
      await new Promise(resolve => setTimeout(resolve, delay));
    }
  }
  throw new Error('Retry exhausted');
}
```

## 6. まとめ

本記事では、LLMを使ったログ異常検知パイプラインの実装方法を紹介しました。従来のルールベースの監視では対応が難しかった、文脈に依存する異常や未知のパターンを検出できる点が最大の利点です。

ただし、LLMによる分析は完璧ではありません。以下の点を考慮して導入を検討してください。

- コストと精度のバランスを取る（全てのログをLLMに送る必要はない）
- プロンプトは定期的に見直し、改善する
- 異常検知の結果は必ず人間が確認できる仕組みを残す

まずは特定のサービスやエラーレベルに絞って導入し、徐々に適用範囲を広げていくことをおすすめします。
