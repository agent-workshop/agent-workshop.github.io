# AIコーディングアシスタントの精度を上げるプロンプト設計

## 1. 導入（なぜこれが必要か、何ができるか）

近年、GitHub CopilotやCursor、Claude Codeなど、AIによるコード生成支援ツールが急速に普及しています。これらは単なるコード補完を超え、関数の実装、リファクタリング、テストコードの生成までこなせるようになりました。しかし、実際に使ってみると「意図と違うコードが返ってくる」「無駄なコードを生成してしまう」といった課題に直面することも多いでしょう。

その原因の多くは、**プロンプト（指示）の設計**にあります。AIは人間のように「文脈を察する」のが得意ではありません。曖昧な指示には曖昧な回答が返ってきます。逆に、適切に設計されたプロンプトを使えば、AIの出力精度は格段に向上します。

本記事では、TypeScript/JavaScriptの開発現場で実際に使えるプロンプト設計のテクニックを、コードサンプルを交えて解説します。「AIに任せたいけど品質が不安」という中級エンジニアの方に、具体的な指針を提供します。

## 2. 前提条件（環境・ライブラリ）

本記事のコードサンプルは、以下の環境で動作確認しています。

- **Node.js**: v18.0.0 以上
- **TypeScript**: v5.0.0 以上
- **使用するAIアシスタント**: GitHub Copilot（VS Code拡張）、または Claude Code（CLI）
- **推奨エディタ**: VS Code（最新版）

特別なライブラリは不要です。TypeScriptの標準的な型定義と、`ts-node` または `tsx` で実行可能な環境を想定しています。

## 3. 実装手順（ステップバイステップ）

プロンプト設計の基本は **「コンテキスト」「制約」「出力形式」** の3要素を明確にすることです。以下の手順で進めます。

### Step 1: コンテキストを明示する
AIに「何を」「なぜ」実装するのかを伝えます。コードだけでなく、使用する技術スタックや既存のコード構造も含めると効果的です。

### Step 2: 制約条件を列挙する
「エラーハンドリングは必須」「パフォーマンスを重視」「特定のライブラリを使わない」など、守るべきルールを箇条書きで指定します。

### Step 3: 出力形式を指定する
「TypeScriptの型定義を含めて」「コメントはJSDoc形式で」「関数は1つだけ」といった具体的なフォーマットを指示します。

### Step 4: 期待する動作例を示す
可能であれば、入力と出力の例を1つ提示します。これによりAIの推論がブレにくくなります。

## 4. コードサンプル

### サンプル1: 関数実装のプロンプト

以下は、AIアシスタントに「非同期処理を含むデータ取得関数」を依頼する例です。

```typescript
// プロンプト:
// 以下の条件で、ユーザー情報を非同期に取得する関数を実装してください。
// - 引数: userId (string)
// - 戻り値: Promise<User | null>
// - エラー時はnullを返し、throwはしない
// - ネットワークエラーはキャッチしてログ出力する
// - キャッシュ機構は不要

interface User {
  id: string;
  name: string;
  email: string;
}

async function fetchUser(userId: string): Promise<User | null> {
  try {
    const response = await fetch(`https://api.example.com/users/${userId}`);
    if (!response.ok) {
      console.error(`HTTP error: ${response.status}`);
      return null;
    }
    const user: User = await response.json();
    return user;
  } catch (error) {
    console.error('Network error:', error);
    return null;
  }
}
```

このプロンプトでは、コンテキスト（非同期処理）、制約（throw禁止、エラーハンドリング）、出力形式（型定義含む）をすべて指定しています。AIは迷わず実装できます。

### サンプル2: リファクタリングのプロンプト

既存コードをAIにリファクタリングさせる例です。

```typescript
// プロンプト:
// 以下のコードをリファクタリングしてください。
// 条件:
// - 関数は1つだけに分割しない（現状維持）
// - 変数名は意味のあるものに変更する
// - 早期リターンを使用する
// - TypeScriptの型を明示する

// リファクタリング前
function calc(x, y, type) {
  let result;
  if (type === "add") {
    result = x + y;
  } else if (type === "sub") {
    result = x - y;
  } else if (type === "mul") {
    result = x * y;
  } else {
    result = 0;
  }
  return result;
}

// リファクタリング後
type OperationType = 'add' | 'sub' | 'mul';

function calculate(x: number, y: number, operation: OperationType): number {
  if (operation === 'add') return x + y;
  if (operation === 'sub') return x - y;
  if (operation === 'mul') return x * y;
  return 0;
}
```

制約条件を細かく指定することで、AIの「やりすぎ」を防ぎつつ、品質を向上させています。

### サンプル3: テストコード生成のプロンプト

テストコードを生成させる際の例です。テストフレームワークはVitestを想定しています。

```typescript
// プロンプト:
// 以下の関数に対する単体テストをVitestで書いてください。
// 条件:
// - 正常系と異常系（空文字、null、undefined）をカバーする
// - describe/it を使用する
// - expect は toBe と toThrow のみ使う

export function greet(name: string): string {
  if (!name) {
    throw new Error('Name is required');
  }
  return `Hello, ${name}!`;
}

// 生成されたテストコード
import { describe, it, expect } from 'vitest';
import { greet } from './greet';

describe('greet', () => {
  it('should return greeting with name', () => {
    expect(greet('Alice')).toBe('Hello, Alice!');
  });

  it('should throw error for empty string', () => {
    expect(() => greet('')).toThrow('Name is required');
  });

  it('should throw error for null', () => {
    expect(() => greet(null as unknown as string)).toThrow('Name is required');
  });

  it('should throw error for undefined', () => {
    expect(() => greet(undefined as unknown as string)).toThrow('Name is required');
  });
});
```

テストケースの粒度を指定することで、網羅的かつ無駄のないテストコードが生成されます。

## 5. ハマりやすいポイント

### ポイント1: プロンプトが長すぎると逆効果
詳細すぎるプロンプトはAIの「注意」を分散させ、かえって品質が低下することがあります。特に、複数のタスクを1つのプロンプトに詰め込むのは避けましょう。1プロンプト＝1タスクが基本です。

### ポイント2: 「暗黙の了解」を前提にしない
人間同士なら「普通、例外は投げるよね」と暗黙に理解されることも、AIには伝わりません。「エラーハンドリングはどうするか」「nullを許容するか」は明示的に指定しましょう。

### ポイント3: 生成されたコードをそのまま信用しない
AIが生成したコードは、あくまで「たたき台」です。型チェックやlintを通し、意図通りに動作するか必ず確認してください。特にセキュリティに関わる処理（認証、バリデーション）は人間のレビューが必須です。

### ポイント4: 同じプロンプトでも結果が変わる
AIモデルは確率的に動作するため、同じプロンプトでも異なる結果が返ることがあります。重要なコードは複数回生成させ、比較検討することをおすすめします。

## 6. まとめ

AIコーディングアシスタントの精度を上げるには、プロンプト設計が鍵です。本記事で紹介した3つの要素「コンテキスト」「制約」「出力形式」を意識するだけで、生成されるコードの品質は大きく変わります。

- **コンテキスト**: 技術スタックや既存コードを明示する
- **制約**: エラーハンドリングやパフォーマンス要件を具体的に列挙する
- **出力形式**: 型定義やコメント形式を指定する

最初はプロンプトを書く手間が増えるかもしれませんが、修正や再生成の回数が減るため、トータルの開発時間は短縮できます。ぜひ日々の開発に取り入れてみてください。
