# AIエージェントのエラーハンドリング設計：サーキットブレーカーパターン

## 1. 導入（なぜこれが必要か、何ができるか）

AIエージェントを実運用する際、外部API（OpenAI、Anthropic、Google AIなど）へのリクエストが失敗することは避けられません。レート制限、一時的なサーバーダウン、ネットワーク障害など、原因は多岐にわたります。

これらの障害に対して、単純なリトライ処理だけでは不十分です。連続して失敗するリクエストがシステム全体を圧迫し、結果的に応答時間の悪化やコスト増加を招く可能性があります。

サーキットブレーカーパターンは、この問題を解決するための設計手法です。障害が一定回数発生した場合に、後続のリクエストを即座に失敗させ（回路を遮断）、システムへの負荷を軽減します。その後、一定時間経過後にリクエストを再開し、復旧を確認します。

この記事では、TypeScriptで実際に動作するサーキットブレーカーの実装を通じて、AIエージェントの堅牢なエラーハンドリング設計を学びます。

## 2. 前提条件（環境・ライブラリ）

- Node.js 18以上
- TypeScript 5.0以上
- 使用するライブラリ: 標準モジュールのみ（外部ライブラリ不要）

本記事では、OpenAI APIを想定したサンプルコードを提供しますが、APIキーがなくても動作するようにモック関数を使用します。

## 3. 実装手順（ステップバイステップ）

サーキットブレーカーの実装は以下の3つの状態を持ちます：

1. **CLOSED（閉状態）**: 通常の動作。リクエストを実行し、失敗回数をカウント
2. **OPEN（開状態）**: 障害発生中。リクエストを即座に拒否
3. **HALF_OPEN（半開状態）**: 復旧確認中。限定的にリクエストを許可

実装の流れは次の通りです：

1. サーキットブレーカーの状態管理クラスを作成
2. リクエスト実行関数にブレーカー処理を組み込む
3. エラー発生時の状態遷移ロジックを実装
4. 実際のAI API呼び出しに適用

## 4. コードサンプル

### サンプル1: 基本のサーキットブレーカー実装

```typescript
type CircuitState = 'CLOSED' | 'OPEN' | 'HALF_OPEN';

interface CircuitBreakerConfig {
  failureThreshold: number;  // 失敗許容回数
  successThreshold: number;  // 復旧成功回数
  timeout: number;           // 開状態からの復帰時間（ms）
}

class CircuitBreaker {
  private state: CircuitState = 'CLOSED';
  private failureCount = 0;
  private successCount = 0;
  private lastFailureTime = 0;
  private nextAttemptTime = 0;

  constructor(private config: CircuitBreakerConfig) {}

  async call<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === 'OPEN') {
      if (Date.now() > this.nextAttemptTime) {
        this.state = 'HALF_OPEN';
      } else {
        throw new Error('Circuit breaker is OPEN. Request rejected.');
      }
    }

    try {
      const result = await fn();
      this.onSuccess();
      return result;
    } catch (error) {
      this.onFailure();
      throw error;
    }
  }

  private onSuccess(): void {
    this.failureCount = 0;
    if (this.state === 'HALF_OPEN') {
      this.successCount++;
      if (this.successCount >= this.config.successThreshold) {
        this.state = 'CLOSED';
        this.successCount = 0;
      }
    }
  }

  private onFailure(): void {
    this.lastFailureTime = Date.now();
    this.failureCount++;
    if (this.failureCount >= this.config.failureThreshold) {
      this.state = 'OPEN';
      this.nextAttemptTime = Date.now() + this.config.timeout;
    }
  }

  getState(): CircuitState {
    return this.state;
  }
}
```

### サンプル2: AIエージェントへの適用

```typescript
// モックのAI API関数（実際のAPIキー不要）
async function callAIAPI(prompt: string): Promise<string> {
  // 50%の確率でエラーをシミュレート
  if (Math.random() < 0.5) {
    throw new Error('API request failed: Rate limit exceeded');
  }
  return `Response for: ${prompt}`;
}

// サーキットブレーカーの設定
const breaker = new CircuitBreaker({
  failureThreshold: 3,   // 3回連続失敗で遮断
  successThreshold: 2,   // 2回連続成功で復旧
  timeout: 10000,        // 10秒後に再試行
});

// AIエージェントのリクエスト処理
async function processAIRequest(prompt: string): Promise<string> {
  try {
    const response = await breaker.call(() => callAIAPI(prompt));
    console.log(`[${new Date().toISOString()}] Success: ${response}`);
    return response;
  } catch (error) {
    if (error instanceof Error && error.message.includes('Circuit breaker is OPEN')) {
      console.warn(`[${new Date().toISOString()}] Circuit breaker OPEN - skipping request`);
      return 'Service temporarily unavailable. Please try again later.';
    }
    console.error(`[${new Date().toISOString()}] API error:`, error);
    throw error;
  }
}

// 実行例
async function main() {
  const prompts = ['質問1', '質問2', '質問3', '質問4', '質問5'];
  
  for (const prompt of prompts) {
    console.log(`\nProcessing: ${prompt}`);
    console.log(`Current state: ${breaker.getState()}`);
    
    try {
      const result = await processAIRequest(prompt);
      console.log(`Result: ${result}`);
    } catch (error) {
      console.error(`Failed: ${error}`);
    }
    
    // リクエスト間隔をシミュレート
    await new Promise(resolve => setTimeout(resolve, 500));
  }
}

main().catch(console.error);
```

### サンプル3: 指数バックオフ付きリトライとの組み合わせ

```typescript
async function callWithRetryAndBreaker(
  prompt: string,
  maxRetries: number = 3,
  baseDelay: number = 1000
): Promise<string> {
  let lastError: Error | null = null;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await breaker.call(() => callAIAPI(prompt));
    } catch (error) {
      lastError = error as Error;
      
      if (error instanceof Error && error.message.includes('Circuit breaker is OPEN')) {
        // サーキットブレーカーが開いている場合は即座に終了
        throw error;
      }

      if (attempt < maxRetries) {
        const delay = baseDelay * Math.pow(2, attempt); // 指数バックオフ
        console.log(`Retry ${attempt + 1}/${maxRetries} after ${delay}ms`);
        await new Promise(resolve => setTimeout(resolve, delay));
      }
    }
  }

  throw lastError || new Error('All retries failed');
}
```

## 5. ハマりやすいポイント

### 状態管理の同期問題
サーキットブレーカーは複数のリクエストから同時にアクセスされる可能性があります。上記の実装は単一スレッドのNode.js環境では問題ありませんが、マルチスレッド環境や分散システムでは排他制御が必要です。その場合は、Redisやデータベースを使って状態を共有する設計を検討してください。

### タイムアウト設定の重要性
サーキットブレーカーの`timeout`値は、実際のAPIの応答時間に基づいて適切に設定する必要があります。短すぎると正常なリクエストまで遮断してしまい、長すぎると障害時の影響が大きくなります。初期値として30秒〜60秒から始め、運用データに基づいて調整することを推奨します。

### HALF_OPEN状態のテスト戦略
HALF_OPEN状態では、1リクエストのみ許可する実装が一般的です。しかし、実際のシステムでは複数のリクエストが同時にHALF_OPENを検出し、同時にAPIを呼び出す可能性があります。これを防ぐには、HALF_OPEN遷移時にアトミックなフラグ管理を行うか、リクエストキューを導入する必要があります。

### ログと監視の不足
サーキットブレーカーの状態遷移を適切にログに記録しないと、システムの動作がブラックボックス化します。状態変更時には必ずログを出力し、メトリクスとして収集することを習慣づけましょう。

## 6. まとめ

サーキットブレーカーパターンは、AIエージェントの堅牢性を向上させるための実践的な手法です。本記事で実装したコードは、以下の特徴を持ちます：

- 外部APIの障害からシステム全体を保護
- 指数バックオフリトライとの組み合わせで効率的な復旧
- 状態遷移の明確なロジックによる予測可能な動作

実運用では、さらに以下の点を考慮すると良いでしょう：

- メトリクス収集（失敗率、平均応答時間など）
- 動的な閾値調整（機械学習による適応）
- サーキットブレーカーの状態を外部から監視・操作できる管理画面

AIエージェントを本番運用する際には、今回紹介したサーキットブレーカーパターンを必ず組み込むことをお勧めします。エラーハンドリングの設計次第で、システムの信頼性は大きく変わります。
