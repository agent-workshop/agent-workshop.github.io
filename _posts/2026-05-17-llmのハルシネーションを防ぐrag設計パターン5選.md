# LLMのハルシネーションを防ぐRAG設計パターン5選

## 1. 導入

LLM（大規模言語モデル）の活用が進む一方で、事実と異なる情報を生成する「ハルシネーション」は実務上の大きな課題です。特に社内文書や製品マニュアルを参照するRAG（Retrieval-Augmented Generation）システムでは、誤った回答が業務に直接影響を及ぼすリスクがあります。

本記事では、RAGシステムにおけるハルシネーションを抑制するための設計パターンを5つ紹介します。各パターンは実際のプロジェクトで検証済みで、TypeScript/JavaScriptでの実装コードを交えて解説します。

## 2. 前提条件

以下の環境を想定します。

- Node.js 18以上
- TypeScript 5.0以上
- OpenAI API（または互換API）
- ベクトルデータベース（PostgreSQL + pgvector、またはPinecone等）

必要なライブラリは以下です。

```bash
npm install openai langchain @langchain/core @langchain/openai
npm install -D typescript @types/node ts-node
```

## 3. 実装手順

### 全体の流れ

1. ドキュメントのチャンク分割とベクトル化
2. 検索クエリの最適化（クエリ拡張）
3. 取得コンテキストのフィルタリング
4. プロンプト設計による制約
5. 回答の事実検証（ファクトチェック）

## 4. コードサンプル

### パターン1: クエリ拡張による検索精度向上

単一の質問だけでは関連情報が検索できないケースがあります。元のクエリから複数のバリエーションを生成し、それぞれ検索することで網羅性を高めます。

```typescript
import { OpenAI } from "@langchain/openai";

async function expandQuery(originalQuery: string): Promise<string[]> {
  const llm = new OpenAI({
    modelName: "gpt-4o-mini",
    temperature: 0.0,
  });

  const prompt = `以下の質問を検索に適した形に3通りに言い換えてください。各バリエーションは改行で区切って出力してください。

元の質問: ${originalQuery}

出力形式（例）:
バリエーション1: ...
バリエーション2: ...
バリエーション3: ...`;

  const response = await llm.invoke(prompt);
  const lines = response.split("\n").filter((line) => line.includes(":"));
  return lines.map((line) => line.split(":")[1].trim());
}

// 使用例
const query = "AWS Lambdaのタイムアウト時間の上限は？";
const expanded = await expandQuery(query);
console.log(expanded);
// ["AWS Lambda タイムアウト 最大値", "Lambda関数 実行時間 制限", "AWS Lambda 設定可能なタイムアウト"]
```

### パターン2: コンテキストのスコアリングとフィルタリング

検索結果をそのままLLMに渡すのではなく、関連性スコアに基づいて不要な情報を除外します。

```typescript
interface DocumentWithScore {
  content: string;
  score: number;
  source: string;
}

function filterContextByScore(
  documents: DocumentWithScore[],
  threshold: number = 0.7
): DocumentWithScore[] {
  // スコアでソート
  const sorted = documents.sort((a, b) => b.score - a.score);

  // 閾値以上のドキュメントのみ採用
  const filtered = sorted.filter((doc) => doc.score >= threshold);

  // 最大3件までに制限
  return filtered.slice(0, 3);
}

// 使用例（Pinecone等のベクトル検索結果を想定）
const searchResults: DocumentWithScore[] = [
  { content: "Lambdaのタイムアウトは最大900秒", score: 0.92, source: "aws_docs_v2" },
  { content: "Lambdaの同時実行数は1000", score: 0.45, source: "aws_docs_v2" },
  { content: "Lambdaのメモリは最大10240MB", score: 0.88, source: "aws_docs_v3" },
];

const validContext = filterContextByScore(searchResults, 0.7);
// score 0.92 と 0.88 の2件のみ採用
```

### パターン3: プロンプトによる回答制約

LLMに「知らないことは知らない」と言わせる設計をプロンプトに組み込みます。

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage, SystemMessage } from "@langchain/core/messages";

async function answerWithContext(context: string, question: string): Promise<string> {
  const chat = new ChatOpenAI({
    modelName: "gpt-4o-mini",
    temperature: 0.2,
  });

  const systemPrompt = `あなたは社内文書に基づいて回答するアシスタントです。
以下のルールに従ってください：

1. 提供されたコンテキスト内の情報のみを使って回答すること
2. コンテキストに該当する情報がない場合は「該当する情報が文書に見つかりませんでした」と正直に回答すること
3. 推測や創作をしてはいけません
4. 回答の最後に参照した文書のソースを明記すること`;

  const messages = [
    new SystemMessage(systemPrompt),
    new HumanMessage(`コンテキスト:
${context}

質問: ${question}`),
  ];

  const response = await chat.invoke(messages);
  return response.content.toString();
}

// 使用例
const context = `AWS Lambdaのタイムアウトは最大900秒（15分）です。
メモリは128MBから10240MBまで設定可能です。`;

const question = "Lambdaのディスク容量は？";
const answer = await answerWithContext(context, question);
// → "該当する情報が文書に見つかりませんでした"
```

## 5. ハマりやすいポイント

### チャンク分割の粒度
チャンクサイズが大きすぎると、無関係な情報が混ざってハルシネーションの原因になります。逆に小さすぎると文脈が失われます。経験則として512〜1024トークンが適切です。

### スコア閾値の調整
フィルタリングの閾値（0.7など）は、プロジェクトのデータ分布に応じて調整が必要です。初めは0.5から始め、実際の回答品質を見ながら徐々に上げていくと良いでしょう。

### プロンプトの温度パラメータ
回答生成時のtemperatureは0.2以下に設定することを推奨します。0.5以上だと創造性が高まり、ハルシネーションが発生しやすくなります。

### ソースの明記
回答に参照元を明記しないと、ユーザーが情報の正確性を検証できません。プロンプトに「必ずソースを記載する」という指示を入れることが重要です。

## 6. まとめ

本記事ではRAGシステムにおけるハルシネーション対策として、以下の5つの設計パターンを紹介しました。

1. **クエリ拡張** - 検索の網羅性を高める
2. **スコアフィルタリング** - ノイズを除去する
3. **プロンプト制約** - 推測を禁止する
4. **チャンク最適化** - 適切な粒度で情報を分割する
5. **ソース明記** - 検証可能性を担保する

これらのパターンを組み合わせることで、ハルシネーションの発生率を大幅に低減できます。ただし、完全な防止は難しいため、出力結果のモニタリングと継続的な改善が欠かせません。
